JUnit 5 Readiness Knowledge Base

Guidance for using EMP JUnit 5 readiness evidence during JUnit 4 to JUnit 5 modernization planning.

Benchmarks3

Guide

  • Start with applicability. The JUnit 5 readiness pack is useful only when JUnit 4 tests or dependencies are detected.
  • Separate test modernization from framework modernization. JUnit 5 work often happens alongside Spring Boot, Java LTS, and build-tool upgrades, but it has its own behavior risks.
  • Treat static findings as triage evidence. They identify test APIs and runner assumptions before a consultant promises migration execution.

Common Risks

RiskWhy It MattersRecommendation
JUnit 4 API usage
junit4-api-usage		 JUnit 4 annotations and assertions need explicit migration to Jupiter APIs or compatibility handling. Inventory JUnit 4 imports and migrate tests in batches with validation evidence.
JUnit 4 runners
junit4-runner-usage		 RunWith-based tests often encode Spring, Mockito, parameterized, or custom runner behavior. Review each runner and replace it with the correct Jupiter extension or test style.
JUnit 4 dependency
junit4-dependency		 A direct JUnit 4 dependency can keep old tests compiling while hiding incomplete migration work. Track dependency replacement separately from source import rewrites.
Vintage engine
junit-vintage-engine		 Vintage can be useful temporarily, but it can also mask the amount of JUnit 4 still in production test suites. Use Vintage deliberately as a transition bridge and report remaining JUnit 4 tests.

Validation Notes

  • Catalog-backed reports prove report shape and risk vocabulary, not checkout compile success.
  • A passing test run is the most important validation evidence for test modernization.
  • Compile-only success is not enough because runner and extension behavior can fail at test execution time.
  • Failed validation evidence is still useful when it identifies build-tool, plugin, or runtime blockers.

What This Does Not Prove

  • The readiness report does not automatically migrate tests.
  • The readiness report does not prove test behavior is preserved.
  • The readiness report does not replace focused test execution after migration.
  • The readiness report does not infer custom runner semantics.
