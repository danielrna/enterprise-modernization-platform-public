Spring Security 6 Readiness Knowledge Base

Guidance for using EMP Spring Security 6 readiness evidence during Spring Security 5.x to 6.x modernization planning.

Risks5

Guide

  • Start with applicability. The Spring Security 6 readiness pack is useful only when Spring Security dependencies or configuration code are detected.
  • Separate security migration from general Boot migration. Jakarta namespace work, Spring Boot 3 compatibility, authorization DSL changes, and method security changes should be tracked as related but distinct workstreams.
  • Treat static security findings as triage evidence. They identify configuration areas that need review before a consultant promises upgrade execution.

Common Risks

RiskWhy It MattersRecommendation
WebSecurityConfigurerAdapter usage
spring-security-websecurityconfigureradapter		 WebSecurityConfigurerAdapter is removed in modern Spring Security configuration styles. Replace adapter-based configuration with SecurityFilterChain beans and validate compile/test evidence.
Legacy matcher APIs
spring-security-legacy-matchers		 antMatchers, mvcMatchers, and regexMatchers usage usually needs explicit request matching review. Review URL matching semantics before replacing matcher APIs.
authorizeRequests configuration
spring-security-authorize-requests		 Authorization DSL changes can alter request authorization behavior if migrated mechanically. Review authorization rules with compile, test, and route-level validation evidence.
Global method security
spring-security-global-method-security		 Method security annotation changes can affect service authorization behavior. Review method security configuration and protect it with focused tests.
javax and Jakarta overlap
javax-usage		 Spring Security 6 work usually happens alongside Spring Boot 3 and Jakarta namespace migration. Track namespace work separately so security behavior risk is not hidden behind import rewrites.

Validation Notes

  • Catalog-backed reports prove the report shape and risk vocabulary, not checkout compile success.
  • Security configuration changes need focused tests for authorization behavior, not only successful compilation.
  • Passing compile/test validation increases confidence, but route-level and method-level access behavior still needs project-specific review.
  • Failed validation evidence is useful when it honestly records build compatibility, runtime, dependency, or test environment blockers.

What This Does Not Prove

  • The readiness report does not prove authorization behavior is preserved.
  • The readiness report does not replace route-level access testing.
  • The readiness report does not guarantee method security semantics are unchanged.
  • The readiness report does not apply Spring Security source transformations.
