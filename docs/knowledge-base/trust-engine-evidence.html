Trust Engine Evidence Guide

Guidance for explaining EMP confidence factors, validation evidence, and report limits during modernization planning.

Packspring-boot-3-readiness
Risks5
Benchmarks5

Guide

  • Start with the confidence score, then explain the factors behind it. A high score should be backed by checkout source, validation status, finding quality, and applicable pack evidence.
  • Use factor status and impact to keep the conversation concrete. Positive factors explain what is proven; negative factors explain what must be fixed before execution confidence increases.
  • Tie every recommendation to an evidence type. Static findings, dependency metadata, checkout-backed validation, and enterprise rules answer different client questions.

Common Risks

RiskWhy It MattersRecommendation
Catalog-only evidence
benchmark-source		 Catalog reports prove repeatable report generation and risk vocabulary, but they do not prove the public checkout still builds. Use catalog-only reports for discovery and comparison, then promote serious client references to checkout-backed validation.
Validation missing
validation-status		 Without compile or test evidence, the report cannot distinguish source-level risk from build-system or environment blockers. Run validation before treating a modernization estimate as execution-ready.
Partial validation
validation-failed		 A compile pass with failing tests is stronger than no validation, but it still leaves runtime confidence unresolved. Preserve the failed test evidence and plan dependency, environment, or service repair as the next paid step.
Pack mismatch
pack-applicability		 A report can be technically valid while not applying to the selected modernization pack. Use the mismatch as scoping evidence and switch to the pack that matches detected dependencies and source patterns.
Enterprise rule violations
enterprise-rules		 Client-owned rules can change migration priority even when framework readiness looks acceptable. Review owner, severity, rationale, and remediation metadata before agreeing on a migration sequence.

Validation Notes

  • Passing compile and test validation increases confidence, but it does not prove production behavior or performance.
  • Failed validation evidence should keep command, duration, exit code, timeout status, and a short output excerpt so the blocker can be explained.
  • Trust factors are conversation aids, not legal guarantees. They make the evidence model explicit enough for a consultant to defend scope.
  • The strongest report combines applicable pack findings, checkout-backed source, passing validation, and client-specific enterprise rules.

What This Does Not Prove

  • Trust Engine evidence does not replace a client repository validation run.
  • A passing public benchmark does not prove a different application will migrate cleanly.
  • Static findings do not prove runtime behavior, data correctness, or production performance.
  • Enterprise rule scoring depends on the client maintaining accurate rule metadata.
